# Expected Result

Пользователь умеет читать базовый отчёт Codex и понимает, как отличить успешный run от остановки, какие файлы были изменены и что делать дальше.

# Ready Answer

## Table of Fields

| Поле | Где искать | Что означает |
| --- | --- | --- |
| `RESULT` | строка `RESULT:` | Успех, стоп или ошибка |
| `TASK_ID` | строка `TASK_ID:` | Идентификатор конкретной задачи |
| `files_changed` | список файлов | Что реально изменилось |
| `tests` | блок `tests:` | Чем результат проверили |
| `commit` | блок `commit:` | Каким git-коммитом закрепили результат |
| `risks` | блок `risks:` | Что может быть рискованным или не до конца готовым |
| `next step` | блок `next step:` | Что делать дальше |

## Worked Example

Пример отчёта:

```text
CHATGPT_REPORT_BEGIN
RESULT: SUCCESS
TASK_ID: OPENSCRIPT_SITE_EXAMPLE_001
FILES_CHANGED:
- source/app/materials/course_content/course.yaml
- source/app/materials/course_content/lessons/01-kak-my-rabotaem.md
TESTS:
- targeted_tests_result: 5 passed
COMMIT:
- a1b2c3d4
NEXT:
- prepare lesson 2
CHATGPT_REPORT_END
```

Как читать этот пример:

- `RESULT: SUCCESS` означает, что run завершился успешно.
- `TASK_ID` подтверждает, что отчёт относится к конкретной задаче.
- `FILES_CHANGED` показывает, какие файлы реально были затронуты.
- `TESTS` показывает, что результат проверили тестом.
- `COMMIT` показывает, что итог закреплён в git.
- `NEXT` показывает, какой шаг должен идти дальше.

# Explanation

Этот пример показывает не просто список строк, а логику проверки:

1. Сначала смотрим статус run.
2. Потом смотрим, какие файлы изменились.
3. Потом смотрим тесты.
4. Потом смотрим commit.
5. Потом смотрим следующий шаг.

Если `RESULT` не `SUCCESS`, нельзя молча идти дальше. Нужно сначала понять причину остановки.

# Common Wrong Answers

- "RESULT просто красиво написан, можно не смотреть."
- "TASK_ID не нужен, если отчёт и так понятен."
- "`files_changed` можно пропустить, если тесты зелёные."
- "Commit не важен, если всё выглядит хорошо."
- "Если есть ошибка, можно сразу просить следующий урок."

# Pass/Fail Checklist

- PASS, если `RESULT` найден и понятен.
- PASS, если `TASK_ID` найден и совпадает с задачей.
- PASS, если `FILES_CHANGED` показывает реальные изменения.
- PASS, если `TESTS` показывает проверку.
- PASS, если `COMMIT` найден.
- PASS, если `NEXT` найден.
- FAIL, если пользователь не может объяснить, что произошло.
- FAIL, если пользователь не может назвать следующий шаг.

# What to do if failed

Если не получилось прочитать отчёт:

1. Сначала найди `RESULT`.
2. Потом найди `TASK_ID`.
3. Потом найди `FILES_CHANGED`.
4. Потом найди `TESTS`.
5. Если отчёт неполный, остановись и задай уточняющий вопрос ChatGPT вместо того, чтобы угадывать.
